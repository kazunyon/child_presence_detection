# まもるバス

**まもるバス**は、保育園・幼稚園の送迎バスで、子どもの置き去りを防ぐための安全確認PWAです。

高価な専用機器だけに頼らず、園にあるスマートフォンとQRコードを活用し、職員が毎日行う確認を

**乗車 → 降車 → 車内の目視確認 → 最後尾確認 → 別の職員による確認 → 記録保存**

の順番で、迷わず進められる仕組みを目指しています。

画面は、ITに詳しくない職員でも使いやすいように、大きなボタンと分かりやすい日本語を中心にしています。AIは職員の確認を補助する役割とし、AIの判定だけで安全確認を完了させない設計方針です。

> [!IMPORTANT]
> 現在は開発途中の試作版です。公開ページは画面と基本操作を確認するためのデモであり、実際の送迎業務にはまだ使用できません。

## 公開デモ

[まもるバスを開く](https://kazunyon.github.io/child_presence_detection/)

デモ用のログイン情報です。

- 職員ID：`1`
- PIN：`1234`
- 最後尾QRの代替文字列：`bus-tail-2`
- 第三者確認用PIN：`5678`

GitHub Pagesで公開しているのはReactの画面部分です。FastAPIとSQLiteはGitHub Pages上では動作しないため、公開デモで操作した内容は正式な記録として保存されません。

## 現在できること

### 公開デモでできること

- スマートフォン向けの「ホーム・園児・記録・設定」画面を表示
- デモ用の職員IDとPINによるログイン
- カメラを使ったQRコード読み取り
  - ブラウザがQR自動読取に対応していない場合は、QRの文字列を手入力可能
- 降車確認から、最後尾QR確認、第三者確認までの一連の操作体験
- 最後尾確認時のGPS取得
  - 位置情報を許可した端末のみ
- 園児・職員・車両・便を登録する設定画面の操作体験
- PWAとしての基本設定
  - 対応端末・ブラウザでは、ホーム画面への追加またはインストールが可能
- GitHub ActionsによるGitHub Pagesへの自動公開

### FastAPIを起動した開発環境でできること

- SQLiteへの基本データ保存
- 職員IDとPINの照合
- 園児・職員・車両・便の登録・一覧取得API
- 園児QRによる降車記録の保存
- 最後尾QR確認の日時・担当者・GPS位置の保存
- 環境変数 `DATABASE_URL` を利用できるデータベース接続構成

> フロント画面には固定のデモデータも含まれています。APIに保存したデータが、すべての画面へ自動反映される段階ではありません。

## 現在できないこと

### 最重要：公開PWAは操作デモであり、実際の送迎業務には使えません

GitHub Pagesで公開しているまもるバスは、画面と操作手順を確認するためのデモです。画面上で便を開始し、乗車・降車・車内確認・第三者承認まで進められますが、これは安全運用の完成版ではありません。

**園児の安否確認、実際の送迎業務、事故防止の最終確認、法令上必要な記録には使用しないでください。** 必ず職員による目視確認と、園で定めた緊急対応手順を優先してください。

### 1. 実データ保存はまだ開始できません

本番用にRender＋PostgreSQLへ配置する設定ファイル（`render.yaml`）と環境変数の雛形（`.env.example`）は追加済みです。しかし、Render上のサービスとデータベースはまだ作成・接続されていません。

そのため、現在は次の状態です。

- 公開PWAの入力は、ブラウザ上のデモ操作です。
- 便、乗車、降車、最後尾確認、第三者承認は正式なデータベースへ保存されません。
- ページ再読み込み、端末変更、ブラウザ終了でデモの状態は失われます。
- 他の職員のスマートフォンと同じ便の人数・園児状態を共有できません。
- 実保存を始めるには、RenderでWeb ServiceとPostgreSQLを作成し、`DATABASE_URL`、`JWT_SECRET`、`CORS_ORIGINS`、`VITE_API_BASE_URL` を設定する必要があります。

### 2. 個別職員認証・権限管理は未実装です

開発用の職員ID・PINは、画面操作を試すためだけの固定値です。本番の本人確認には使えません。

未実装の内容：

- 職員ごとのアカウント発行・停止・PIN変更
- PINの安全なハッシュ化と再設定手順
- JWT等によるログインセッションとログアウト
- 運転担当、第三者確認者、園管理者の権限分離
- 「運転者本人が第三者承認する」ことを防ぐ厳密な制御
- ログイン失敗、権限外操作、PIN変更の監査ログ

依存ライブラリは追加済みですが、認証・権限機能そのものはこれから実装します。

### 3. 便単位の照合は、公開版では正式な保証になりません

画面には、乗車数・降車数・未降車数を表示し、未降車がいる場合に最後尾確認・承認・便完了を止める操作フローがあります。ただし公開版はデモ保存のため、実際の園児名簿や他端末の記録との正確な照合を保証しません。

また、次は未実装です。

- 欠席、当日変更、途中乗車・途中降車の管理
- 保護者または受入担当者への引渡し確認
- QRコードの発行、印刷、再発行、失効、重複防止
- 複数車両・複数便の配車と職員割当
- 乗降記録の取消し・修正と、修正理由の保存

### 4. 車内確認の証跡は未完成です

- 30秒車内動画の撮影、アップロード、再生、保管はできません。
- AIによる画像・動画の補助判定はできません。
- 最後尾QRとGPSの画面操作はありますが、公開デモでは正式な証跡として保管されません。
- 第三者確認はデモ用PINであり、別人性・承認者・承認時刻を改ざん検知可能な形で保存していません。

### 5. 管理者・保護者への実通知は未接続です

通知キューのAPIはありますが、以下の実送信はまだできません。

- 未確認から一定時間後の自動監視・自動アラーム
- 管理者へのPush通知、メール、SMS、電話連携
- 保護者への降車完了通知
- 送信成功・未達・既読・再通知の追跡
- 園ごとの通知先・連絡時間帯・送信文面の設定

LINE、メール、SMSなどを実送信するには、通知事業者の契約、送信元審査、APIキーの安全な管理が必要です。

### 6. 記録・監査・個人情報保護は未実装です

- 「記録」画面は見本であり、実際の運行履歴の検索画面ではありません。
- PDF/CSV出力、監査用帳票、長期保管、バックアップ、障害復旧は未実装です。
- 通信断時のオフライン記録、復旧後の安全な再送信は未実装です。
- 園児・保護者情報、GPS、動画の暗号化とアクセス制御は未実装です。
- 園ごとのデータ分離（他園の職員が閲覧できない仕組み）は未実装です。
- PostgreSQLへの実接続・移行試験、負荷試験、実園での運用試験は未実施です。

> **実装の優先順**：①Render＋PostgreSQLを実際に作成して保存を有効化 → ②個別職員アカウント・JWT・園ごとのデータ分離 → ③実記録一覧と監査ログ → ④管理者・保護者への実通知 → ⑤動画・AI・オフライン対応、の順で進めます。
## 目指す運用の流れ

1. 園児が乗車したら、園児QRを読み取ります。
2. 到着後、園児が降車するたびにQRを読み取ります。
3. 乗車人数と降車人数が一致したことを確認します。
4. 職員が座席、足元、座席の下、荷物の陰を目視確認します。
5. 車両の最後尾に設置したQRを読み取ります。
6. 車内を30秒撮影し、AIが確認を補助します。
7. 運転担当者とは別の職員が、車内と記録を再確認します。
8. 第三者承認後、日時・GPS・担当者・確認記録を保存します。
9. 必要に応じて、保護者へ降車完了を通知します。

> この流れは完成時の目標です。現時点では、すべての工程が実装済みではありません。

## 安全上の考え方

まもるバスは、職員による安全確認を支援するシステムです。

- アプリの表示だけで「安全」と判断しない
- 必ず職員が車内の最後尾まで移動して目視確認する
- 運転担当者とは別の職員が再確認する
- 人数が一致しない場合は完了扱いにしない
- 園児の所在が分からない場合は、園の緊急対応手順に従う

スマートフォン、QRコード、動画、AIのいずれか一つに頼るのではなく、複数の確認を重ねて見落としを減らすことを目的としています。

## 開発環境

### フロントエンド

- React
- TypeScript
- Vite
- Tailwind CSS
- vite-plugin-pwa

```powershell
npm install
npm run dev
```

### バックエンド

- FastAPI
- Python
- SQLAlchemy
- SQLite（開発用）

```powershell
cd backend
python -m pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

フロントエンドは、環境変数 `VITE_API_BASE_URL` でFastAPIのURLを指定できます。未指定時は `http://127.0.0.1:8000` を使用します。

## GitHub Pagesへの公開

`main` ブランチへ変更を反映すると、GitHub Actionsがフロントエンドをビルドし、GitHub Pagesへ公開します。

初回だけ、GitHubリポジトリの **Settings → Pages → Build and deployment → Source** を **GitHub Actions** に設定してください。
