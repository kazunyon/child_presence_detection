# まもるバス

**まもるバス**は、保育園・幼稚園の送迎バスで、子どもの置き去りを防ぐための安全確認PWAです。

高価な専用機器だけに頼らず、園にあるスマートフォンとQRコードを活用し、職員が毎日行う確認を

**乗車 → 降車 → 車内の目視確認 → 最後尾確認 → 別の職員による確認 → 記録保存**

の順番で、迷わず進められる仕組みを目指しています。

画面は、ITに詳しくない職員でも使いやすいように、大きなボタンと分かりやすい日本語を中心にしています。AIは職員の確認を補助する役割とし、AIの判定だけで安全確認を完了させない設計方針です。

> [!IMPORTANT]
> 現在は開発途中の試作版です。公開ページは画面と基本操作を確認するためのデモであり、実際の送迎業務にはまだ使用できません。

## 公開デモ

[まもるバスを開く](https://kazunyon.github.io/child_presence_detection/)

デモ用のログイン情報です。

- 職員ID：`1`
- PIN：`1234`
- 最後尾QRの代替文字列：`bus-tail-2`
- 第三者確認用PIN：`5678`

GitHub Pagesで公開しているのはReactの画面部分です。FastAPIとSQLiteはGitHub Pages上では動作しないため、公開デモで操作した内容は正式な記録として保存されません。

## 現在できること

### 画面（GitHub Pages）でできること

- スマートフォン向けの「帰り便」画面を表示し、便の開始から完了までの操作を試す
- デモ用PIN（運転担当：`1234`、第三者確認：`5678`）でログイン・第三者承認を試す
- カメラで園児QR・最後尾QRを読み取る
  - `BarcodeDetector` 非対応のブラウザでは、QR文字列を手入力できる
- 乗車数・降車数・未降車数を表示する
- 未降車の園児がいる間は、最後尾確認・第三者承認・便完了の操作を止める
- 最後尾QR（`bus-tail-2`）確認、第三者承認、便完了までの画面フローを試す
- PWAとしてビルド・GitHub Pagesへ自動公開する

公開ページは、GitHub Actionsに `VITE_API_BASE_URL` が設定され、APIへ接続できる場合はAPIの便作成・乗降記録・便完了を利用します。APIに接続できない場合は自動的にブラウザ内のデモ便へ切り替わり、画面を再読み込みすると状態は失われます。

### FastAPIを起動した開発環境でできること

- SQLite（既定）または `DATABASE_URL` で指定したPostgreSQLへ基本データを保存する
- 初期データとして職員、車両、便、園児を作成する
- 園児・職員・車両・便の登録と一覧取得APIを利用する
- 職員IDとPINの照合APIを利用する
- 便を作成し、園児QRによる乗車・降車記録を保存する
- 同じ園児の二重乗車・二重降車や、乗車記録のない降車をAPIで拒否する
- 未降車人数を取得し、未降車がいる便の完了をAPIで拒否する
- 最後尾QRなどの車両安全確認、通知キュー、園児ごとの簡易安全イベントをAPIへ保存する

> APIを起動しただけでは、フロントエンドのすべての画面や操作がAPIと連動するわけではありません。現在の画面から利用しているのは、便作成・乗車／降車・便完了です。

## 現在できないこと

### 最重要：実際の送迎業務には使用できません

本アプリは開発途中の試作版です。公開ページや開発用APIの記録は、園児の安否確認、事故防止の最終確認、法令上必要な記録に使用できません。必ず職員による目視確認と、園で定めた緊急対応手順を優先してください。

### 1. 公開ページでの保存・共有は保証されません

- GitHub Pages自体はバックエンドを持たず、API URLが未設定・未接続の場合はデモ状態だけで動作します。
- RenderとPostgreSQLを使うための定義はありますが、公開ページが実際にどのAPI・データベースへ接続しているかはGitHub Actionsの `VITE_API_BASE_URL` 設定に依存します。
- 複数端末の同時利用、競合、リアルタイム共有、実運用に耐える可用性は未検証です。

### 2. 認証・権限管理は本番仕様ではありません

- 画面のログインは固定PINだけで、APIのログイン結果もセッションやJWTとして保護されません。
- 職員アカウントの発行・停止、PIN変更・再設定、多要素認証、ログアウトはありません。
- 運転担当・第三者確認者・管理者の権限分離はなく、運転者本人による第三者承認を技術的に防止できません。
- ログイン失敗、権限外操作、設定変更などの監査ログはありません。

### 3. 運行・園児管理の範囲が限定的です

- 画面は「帰り便」のみで、園児一覧・記録一覧・設定画面は現在ありません。
- APIの園児・職員・車両・便の登録機能を、画面から操作することはできません。
- 欠席、当日変更、途中乗降、引渡し確認、配車、職員割当、記録の取消し・修正は未実装です。
- QRコードの発行、印刷、再発行、失効、重複利用の防止は未実装です。

### 4. 車内確認の証跡は未完成です

- 最後尾QRの確認は画面上の状態のみで、現在の画面からAPIへ保存していません。
- GPSの取得・保存は画面では実装されていません（APIは緯度・経度を受け取れます）。
- 30秒動画の撮影・アップロード・再生・保管、およびAIによる画像・動画判定はありません。
- 第三者承認は固定PINの画面操作で、承認者・承認時刻・別人性を証跡として保存していません。

### 5. 実通知・監査・個人情報保護は未実装です

- 通知キューAPIはありますが、Push通知、メール、SMS、電話、LINEなどを実送信しません。
- 未確認の自動監視・アラーム、送達・既読・再通知の追跡、園ごとの通知設定はありません。
- 実運行履歴の検索画面、PDF/CSV出力、監査帳票、バックアップ、障害復旧はありません。
- オフライン記録と復旧後の安全な再送信、園ごとのデータ分離、暗号化・アクセス制御、負荷試験・実園での運用試験は未実施です。

> **次の優先事項**：本番用の認証・権限管理、園ごとのデータ分離、実記録の検索と監査ログ、通知連携、オフライン・動画・AI対応を順に整備します。
## 目指す運用の流れ

1. 園児が乗車したら、園児QRを読み取ります。
2. 到着後、園児が降車するたびにQRを読み取ります。
3. 乗車人数と降車人数が一致したことを確認します。
4. 職員が座席、足元、座席の下、荷物の陰を目視確認します。
5. 車両の最後尾に設置したQRを読み取ります。
6. 車内を30秒撮影し、AIが確認を補助します。
7. 運転担当者とは別の職員が、車内と記録を再確認します。
8. 第三者承認後、日時・GPS・担当者・確認記録を保存します。
9. 必要に応じて、保護者へ降車完了を通知します。

> この流れは完成時の目標です。現時点では、すべての工程が実装済みではありません。

## 安全上の考え方

まもるバスは、職員による安全確認を支援するシステムです。

- アプリの表示だけで「安全」と判断しない
- 必ず職員が車内の最後尾まで移動して目視確認する
- 運転担当者とは別の職員が再確認する
- 人数が一致しない場合は完了扱いにしない
- 園児の所在が分からない場合は、園の緊急対応手順に従う

スマートフォン、QRコード、動画、AIのいずれか一つに頼るのではなく、複数の確認を重ねて見落としを減らすことを目的としています。

## 開発環境

### フロントエンド

- React
- TypeScript
- Vite
- Tailwind CSS
- vite-plugin-pwa

```powershell
npm install
npm run dev
```

### バックエンド

- FastAPI
- Python
- SQLAlchemy
- SQLite（開発用）

```powershell
cd backend
python -m pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

フロントエンドは、環境変数 `VITE_API_BASE_URL` でFastAPIのURLを指定できます。未指定時は `http://127.0.0.1:8000` を使用します。

## GitHub Pagesへの公開

`main` ブランチへ変更を反映すると、GitHub Actionsがフロントエンドをビルドし、GitHub Pagesへ公開します。

初回だけ、GitHubリポジトリの **Settings → Pages → Build and deployment → Source** を **GitHub Actions** に設定してください。
